# vibe-check 냉정 평가 & v0.2.0 개선 기록

이 문서는 2026-07-10, vibe-check v0.1.0을 "20년차 시니어 개발자 + AI 자신"의
시선으로 냉정하게 평가한 내용과, 그 결과로 나온 v0.2.0 개선 전체를 정리한
기록이에요. 왜 이렇게 만들었는지 나중에 잊지 않기 위한 설계 근거 문서입니다.

---

## 1. 출발점 — AI에게 코드를 안 보고 못 맡기는 진짜 이유

AI는 "완성된 상태"가 아니라 **"완성된 것처럼 보이는 상태"를 최적화**한다.
구체적으로 자주 저지르는 실수:

1. **에러 삼킴** — 데모가 돌아가게 만드는 최단 경로가 `try/except pass`라서,
   실패를 고치는 대신 숨긴다
2. **세션 기억 상실 → 중복 재구현** — compact 이후 "이미 있는 함수"를 모르고
   또 만든다
3. **자신감 있는 날조** — API 파라미터명, rate limit, 가격을 훈련 데이터
   기억으로 공식 스펙처럼 말한다
4. **무단 결정** — 애매한 요구사항을 묻지 않고 임의 해석으로 확정한다
5. **자기 코드에 대한 자기 리뷰의 무력함** — 쓸 때 한 착각을 검토할 때도
   똑같이 한다 (→ 실행 검증만이 해독제)

v0.1.0의 문제 정의(README의 5가지)는 1~4번을 정확히 겨냥했다.
**5번이 빠져 있었고, 이것이 v0.2.0의 핵심 추가가 됐다.**

---

## 2. v0.1.0 냉정 평가

### 잘한 것

- **훅의 결정론적 설계** — handover 자동 기록을 모델이 아니라 Python이
  수행. "컨텍스트가 사라지는 순간"은 정확히 모델을 신뢰할 수 없는 순간인데,
  그 방어를 모델 밖으로 뺐다. 최고 수준의 판단.
- **훅의 절제** — 절대 안 막고(exit 0), init 안 한 프로젝트에선 침묵,
  잔소리는 세션당 1회, 주입은 800자 제한. "잔소리 훅은 플러그인 삭제로
  이어지는 지름길"이라는 통찰.
- **증거 계약의 정직함** — "dead candidate ≠ 삭제 판정", "부재 주장엔 스캔
  범위 인용". 도구가 자기 맹점을 스스로 공시.
- **자기 모순의 문서화** — `_common.py`의 "넓은 except는 금지지만 호스트를
  깨지 않는 게 상위 계약인 유일한 지점" 주석.

### 냉정한 단점 (심각도 순)

1. **근본 역설: 감시 대상에게 감시를 맡김.** 4축 중 3축(규율 스킬들)이 AI가
   프롬프트를 자발적으로 따르기를 기대하는 텍스트. 강제력은 훅+lint뿐.
2. **가장 필요한 사람이 가장 안 쓰는 UX.** pre/post-write는 수동 커맨드인데,
   이 게이트가 필요한 사람의 정의가 "언제 의심해야 하는지 모르는 사람".
3. **TypeScript 미지원.** 스캐너가 `.ts/.tsx/.jsx`를 못 읽음 — AI로 웹앱
   만드는 주 타깃층에게 "중복 방지" 축이 사실상 비활성.
4. **문서 축이 스캔 축을 오염.** MODULE.md에 함수 이름이 적히면 그 함수는
   영원히 dead 후보에서 빠짐 — 문서를 성실히 쓸수록 죽은 코드 탐지가
   무뎌지는 자기 간섭.
5. **중복 탐지가 실제 AI 실수를 못 잡음.** AST 완전 일치만 잡는데, AI의
   중복은 복붙이 아니라 "90% 비슷한 재구현".
6. **안전벨트의 침묵 실패.** python3가 없으면 훅 전체가 티도 안 나게 꺼짐 —
   확인 수단 없음.
7. **자기 규율을 자기가 안 지킴.** 사용자에겐 테스트를 요구하면서 스캐너·훅에
   테스트 0개.
8. **검증(실행) 축 부재.** post-write가 정적 체크리스트뿐 — "돌려봤는가"가
   없음 (1장의 5번 실수).

한 줄 요약: *"안전벨트라기보다 안전벨트 착용 안내방송 + 에어백 한 개."*
→ 처방: **프롬프트를 늘리지 말고 기계 층을 두껍게.**

## 3. v0.2.0 개선 7가지 (전부 반영됨)

| # | 개선 | 구현 |
|---|---|---|
| 1 | TS/TSX/JSX 지원 | `audit.py`에 `.ts .tsx .jsx .mts .cts` + class/interface/type/enum 추출, `.d.ts`는 참조 텍스트로만 |
| 2 | 문서/코드 참조 분리 | `.md`/`.css` 언급은 dead 후보 구제 불가, `doc_mentions` 필드로 별도 보고 (삭제 시 같이 고칠 문서 목록) |
| 3 | 에러 삼킴 기계 감지 | PostToolUse 훅 `post_write_guard.py` — 빈 except/catch를 정규식 감지, `vibe-check: allow-swallow`로 의도 표시, Edit은 old 대비 **증가분만** 경고 |
| 4 | `/vibe-check:doctor` | 훅 4종+스캐너를 임시 프로젝트에서 **실제 실행**해 ✅/❌ 보고 — 침묵 실패를 드러냄 |
| 5 | 실행 검증 필수화 | post-write 10항목으로 확장: ⑦ 숨은 결합, ⑨ 실행 검증(필수). 마지막 두 줄 계약 = "실행 검증 + 문서 동기화" |
| 6 | 테스트 + CI | `tests/` 34개 (stdlib unittest), GitHub Actions (Python 3.9/3.12) |
| 7 | 유사 중복 탐지 | 정규화 AST(함수명·데코레이터·docstring·**지역 변수명** 치환) 90% 이상 유사 쌍 = `near_duplicate_functions`. 완전 중복도 변수명 무시로 상향 |

### 도구가 밥값을 한 순간 (FP-02의 탄생)

플러그인 저장소를 자기 스캐너로 스캔 → dead 후보 7개 = 전부 방금 만든
**테스트 클래스**. unittest가 이름 규칙으로 호출하니 참조 0이 정상인 오탐.
→ `test_*.py`/`conftest.py`/`*.test.ts` 등을 dead 판정에서 제외 + 회귀
테스트 추가 + 오탐 인덱스에 FP-02로 기록. 재스캔 결과 오탐 0.
**지인에게 배포했으면 가장 먼저 났을 민원을 자기 검사로 미리 잡았다.**

## 4. "프롬프트가 너무 길지 않나?" — 실측 결론

걱정의 실체는 총량이 아니라 ①한 번에 로드된 지시의 밀도, ②출력 강제
장치의 유무. 실측(2026-07-10):

| 층 | 크기 | 로드 시점 |
|---|---|---|
| 스킬+커맨드 description 15개 | 합계 ~3,400자 (토큰 추정 1.5~2k) | 항상 상주 |
| 스킬 본문 6개 | 각 46~100줄 | 트리거될 때만, 보통 1~2개 |
| SessionStart 주입 | 800자 상한 (compact 후 1,500자) | 세션 시작 1회 |
| 에러 삼킴 경고 | ~290자 | 감지될 때만 |
| 설명서/이 문서 | 0 | LLM에 로드 안 됨 (사람용) |

결론:
- **점진 공개 구조라 총량은 문제없음** (상주분이 컨텍스트의 ~1%).
- 제일 밀도 높은 post-write 10항목은 **항목별 ✅/⚠️ 출력 강제**가 준수
  장치 — "명심해라"는 잊혀도 "도장을 찍어라"는 건너뛰면 티가 난다.
- 유일한 보완: **비례 원칙 추가** — 무관한 항목은 "해당 없음" 한 마디로
  통과(필수 ⑨⑩은 항상). 형식적 도배와 통째 생략을 둘 다 막는다.
- **description은 자르지 말 것** — 한국어 트리거 문구 열거는 장식이 아니라
  스킬 발동을 결정하는 매칭 표면. 여기서 아끼면 토큰 몇백 개 아끼고
  스킬 미발동(=플러그인 무력화)을 산다. 최악의 교환.

## 5. lumin-repo-lens 비교 (2026-07-10 로컬 클론 실측)

같은 철학("스캔 없이 주장 금지")에서 출발한 **다른 종의 도구.**
lumin은 구조 분석 한 축을 극단적으로 깊게, vibe-check는 그 축은 얇게
가져가는 대신 lumin이 안 다루는 세 축(기억·행동 규율·기계 강제)을 덮는다.

| | lumin-repo-lens | vibe-check |
|---|---|---|
| 분석 엔진 | Node.js ~5만 줄 (oxc-parser) | Python stdlib 468줄 |
| 분석 깊이 | 콜그래프, 클론 티어, shape 인덱스, barrel 규율, canon 드리프트 | 이름 참조 카운트, 정규화 AST 중복/유사중복 |
| 언어 | TS/TSX/JS 1급, Python은 L1 ("Python 위주면 비추천"이라 자기 명시) | Python 1급 + JS/TS (단순한 수준으로 동급) |
| 요구사항 | Node ≥ 20.19 | python3만 |
| 훅 | 모든 프롬프트 + 모든 도구 호출에 개입 | Write/Edit + compact/시작/종료만, 절대 비차단 |
| 지시 텍스트 | canonical 2,449줄 + references 1,349줄 | 스킬 본문 합 ~410줄 |
| 세션 기억 / AI 행동 규율 / 린트 설치 | 없음 | 핵심 기능 |
| 성숙도 | 0.9.0-beta.93 (93번의 베타) | 0.2.0 |

세 가지 평가:

1. **구조 분석만 보면 상대가 안 된다.** lumin은 진짜 정적 분석기, 우리
   스캐너는 그 옆에선 청진기. 단 이건 의도된 트레이드오프 — "설치 장벽 0,
   얕지만 한계를 정직하게 공시"가 우리 선택이고, 그 정직성 계약 자체가
   lumin에서 온 유산 (README 크레딧).
2. **lumin은 "긴 프롬프트 리스크"의 실사례.** canonical+references ~3,800줄
   지시 문서와 전방위 훅 — 강력하지만 무겁고, LLM이 다 지키는지가 바로
   그 리스크에 노출. vibe-check의 "규율은 짧게, 강제는 기계로" 분화가
   이 지점에서 옳았다.
3. **93번의 베타가 쌓은 흉터는 배울 것** — false-positive 인덱스가 대표.
   → v0.2.0에서 차용 완료 (아래).

**지인 배포 목적엔 vibe-check가 맞는 물건.** 한국어 입문 바이브코더에게
lumin은 Node 20+ / TS 모노레포 전제 / 영어 위주로 진입장벽이 높고,
지인들이 실제로 겪을 문제(기억 상실, 에러 땜빵, 숫자 환각)는 lumin의
관할 밖. 두 플러그인은 경쟁이 아니라 보완 관계.

### lumin에서 차용한 것 — 오탐 인덱스

`skills/repo-xray/references/false-positive-index.md` 신설. 차용한 건
설계 3가지 (내용은 전부 우리 스캐너의 실제 맹점, 문서에 출처 명시):

- 축약 인덱스만 컨텍스트에 싣기 (긴 원장은 배제 — 컨텍스트 예산 보호)
- 증상 → 패밀리 빠른 매핑 표
- "FP ID는 사용자에게 숨기고 쉬운 한국어 이유로 번역" 채팅 규칙

패밀리 9종: 미사용 6종(데코레이터 등록 / 테스트 파일 / 동적 호출 /
미스캔 파일 표면 / 공개 API / 문자열 배선) + 중복 2종(테스트 유사는 정상 /
보일러플레이트) + 충돌 1종(ESM 모듈 스코프는 충돌 아님).
**살아있는 문서 규칙**: 새 오탐 확인 → 패밀리 추가 + 회귀 테스트.
SKILL에 "판정 제시 전 인덱스 대조"를 의무 단계로 배선.
