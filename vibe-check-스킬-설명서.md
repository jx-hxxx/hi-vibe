# vibe-check 스킬 설명서 — 처음 읽는 사람용

이 문서는 지현님이 만든 Claude Code 플러그인 "vibe-check" 안에 들어 있는
스킬 6개를, 코딩을 잘 모르는 사람도 이해할 수 있게 처음부터 끝까지 풀어서
설명하는 글이에요. 천천히 읽으면 됩니다.

---

## 0. 먼저: "스킬"이라는 게 도대체 뭔가요?

Claude(AI)는 똑똑하지만, 매번 대화를 새로 시작하면 "우리 팀의 일하는 방식"을
모르는 신입사원 같아요. 스킬은 그 신입사원 책상 서랍에 넣어둔 **업무 매뉴얼**이에요.

작동 방식은 이래요. 각 매뉴얼(스킬)의 표지에는 "이런 상황이 오면 나를 펼쳐 읽으세요"
라는 안내문이 붙어 있어요. Claude는 평소에 표지의 안내문만 기억하고 있다가,
지현님이 하는 말이 그 안내문과 맞아떨어지면 — 예를 들어 "에러 나는데 고쳐줘"라고
하면 — 해당 매뉴얼을 서랍에서 꺼내 전체를 읽고, 거기 적힌 규칙대로 일해요.

왜 이렇게 만들었냐면, 모든 규칙을 처음부터 다 읽게 하면 AI의 머릿속(컨텍스트)이
꽉 차서 느려지고 비싸지기 때문이에요. 그래서 "표지만 항상 보이게, 내용은 필요할 때만"
이라는 구조를 씁니다. 이걸 어려운 말로 '점진적 공개'라고 해요.

vibe-check에는 매뉴얼이 6권 있어요:

1. **root-cause-first** — 에러를 땜빵하지 말고 원인을 고치게 하는 매뉴얼
2. **grounded-answers** — 헷갈리면 묻고, 근거 없는 숫자를 말하지 못하게 하는 매뉴얼
3. **write-gate** — 코드를 만들기 전/후에 검문소를 통과하게 하는 매뉴얼
4. **docs-keeper** — 프로젝트 문서 4종을 관리하는 매뉴얼
5. **repo-xray** — 저장소를 스캔해서 "증거"를 뽑아내는 도구 + 사용 매뉴얼
6. **guards-setup** — 규칙을 사람 말이 아니라 기계가 강제하게 만드는 설치 매뉴얼

이제 한 권씩 자세히 봅시다.

---

## 1. root-cause-first — "두통의 원인을 찾아라, 두통약만 먹이지 말고"

### 한 줄 요약
AI가 에러를 만났을 때 임시방편으로 가리지 말고, 진짜 원인을 찾아 고치게
강제하는 매뉴얼이에요.

### 왜 필요한가요?

병원에 갔는데 의사가 원인 검사는 안 하고 "아프면 그냥 진통제 드세요"라고만
하면 어떨까요? 당장은 안 아프겠지만 병은 몸속에서 계속 자라겠죠.

AI가 코딩할 때 정확히 이런 짓을 자주 해요. 에러가 나면 원인을 찾는 대신
"에러가 나면 그냥 무시하고 지나가라"는 코드를 덧붙여요. 그러면 프로그램이
겉으로는 멀쩡히 돌아가는 것처럼 보이는데, 사실은 속에서 계속 실패하고 있어요.
더 무서운 건 — **에러 메시지가 안 뜨니까 사람이 문제가 있다는 사실 자체를
모르게 된다**는 거예요. 지현님이 "피드백 루프를 죽인다"고 표현한 게 바로
이거예요. 문제를 알려주는 신호등을 꺼버리는 거죠.

### 매뉴얼 안에 뭐가 들어 있나요?

**첫째, 핵심 계약 3줄.** 매뉴얼 맨 위에 이렇게 적혀 있어요.

- "증상을 가리는 코드는 수정이 아니다" — 에러 메시지를 안 보이게 만드는 건
  고친 게 아니라 숨긴 거다.
- "원인을 모르면 '모른다'고 말한다" — 모르면서 아는 척 대충 고치지 마라.
- "fallback은 설계 결정이지 응급처치가 아니다" — fallback(대체 동작)은
  처음부터 계획하고 넣는 것이지, 급하니까 붙이는 반창고가 아니다.

**둘째, 금지 패턴 목록.** AI가 쓰면 안 되는 코드 모양을 구체적으로 나열했어요.

- *빈 except / 빈 catch*: "에러가 나면 아무것도 하지 마라"는 코드예요.
  에러를 쓰레기통에 버리는 것과 같아요. 금지.
- *실패를 숨기는 기본값*: 예를 들어 "값을 못 가져오면 그냥 0이라고 쳐라"
  같은 코드요. 주식 시세를 못 가져왔는데 화면에 0원이라고 뜨면, 사용자는
  "이 주식이 0원이구나"라고 잘못 믿게 돼요. 실패는 실패라고 보여야 해요. 금지.
- *크래시만 멎게 하는 가드*: 프로그램이 죽으니까 "그 값이 없으면 그냥
  돌아가라"는 문장을 앞에 붙이는 것. 왜 값이 없는지는 안 찾고요. 금지.
- *로그 없는 재시도*: 실패하면 몰래 다시 시도하는 코드. 다시 시도하는 건
  괜찮지만, "실패했었다"는 기록(로그)은 반드시 남겨야 해요.
- *테스트 통과용 방어코드*: 테스트가 자꾸 빨간불이 뜨니까, 테스트를 속이려고
  실제 코드에 예외 처리를 덕지덕지 붙이는 것. 시험 문제를 풀지 않고 답안지를
  고치는 행위예요. 금지.
- *호출부마다 중복 방어*: 같은 걱정을 100군데에서 반복하는 것. 문단속은
  건물 입구(경계)에서 한 번 하는 거지, 방마다 자물쇠를 20개씩 달지 않아요.

**셋째, fallback 허용 시험.** fallback을 정말 쓰고 싶으면 세 가지 질문을
모두 통과해야 해요. 하나라도 "아니오"면 못 써요.

1. 요구사항에 "실패해도 서비스는 계속되어야 한다"고 명시되어 있는가?
   (예: 시세 서버가 죽어도 화면은 떠야 한다 — 이건 정당한 설계)
2. 그 실패가 외부에서 오는, 예상 가능한 실패인가? (네트워크 끊김, 외부 API
   장애, 사용자의 이상한 입력 — 이런 건 내 잘못이 아니라 대비하는 게 맞음)
3. fallback이 작동해도 "지금 실패 중"이라는 사실이 로그와 화면에 드러나는가?

**넷째, 버그 수정 5단계 순서.** AI가 버그를 고칠 때 반드시 밟아야 하는 순서예요.

1. 재현한다 — 버그를 내 눈으로 다시 일으켜본다. 재현을 못 하면 고쳤는지
   확인할 방법도 없으니까요.
2. 위치를 특정한다 — 에러 메시지와 로그를 읽고 "정확히 어디서" 터졌는지
   찾는다. 감으로 고치지 않는다.
3. 원인을 고친다 — 에러가 나타난 지점이 아니라, 에러를 만든 지점을 고친다.
4. 청소한다 — 원인을 찾느라 임시로 넣었던 출력문 같은 걸 지운다.
5. 검증한다 — 1번에서 만든 재현 방법으로 다시 해봐서 진짜 고쳐졌는지 확인한다.

**다섯째, 정직한 항복 규칙.** 이게 제일 중요한 장치예요. 원인 찾기를 2번
시도해서 다 실패하면, AI는 "일단 에러 무시 코드로 감싸고 되나 보자" 하고
도망치는 게 아니라, **멈춰서 사용자에게 솔직하게 보고**해야 해요 — 지금까지
확인한 것, 아닌 걸로 밝혀진 가설, 다음에 확인해볼 것. 왜 이 규칙이 필요하냐면,
AI에게 "실패해도 된다"는 출구를 만들어주지 않으면, AI는 체면을 지키려고
땜빵으로 도망가거든요.

## 2. grounded-answers — "헷갈리면 물어보고, 숫자는 근거를 대라"

### 한 줄 요약
AI가 ①애매한 결정을 제멋대로 내려버리는 것과 ②확인도 안 한 숫자를 사실처럼
말하는 것, 이 두 가지 버릇을 고치는 매뉴얼이에요. (지현님이 직접 추가 요청한
두 가지가 이 스킬이 됐어요.)

### Part 1 — 언제 물어봐야 하는가

이 부분을 설계할 때 제일 고민한 게 있어요. "헷갈리면 무조건 물어봐"라고만
써놓으면, AI가 "변수 이름은 a로 할까요 b로 할까요?" 같은 것까지 전부 물어봐서
그것대로 지옥이 돼요. 그래서 **물어야 할 때와 그냥 진행해야 할 때의 경계선**을
그었어요.

**반드시 물어야 하는 경우 4가지:**

- 해석이 여러 개고, 어느 쪽이냐에 따라 **사용자가 보는 결과가 달라질 때.**
  예: "정렬해줘"라는 말이 가나다순인지 최신순인지 — 화면이 완전히 달라지죠.
- 데이터 구조나 화면 흐름을 좌우하는 **요구사항이 아예 빠져 있을 때.**
  예: 회원가입을 만드는데 이메일 인증을 할지 말지 아무 말이 없었다면.
- **되돌리기 어렵거나 파괴적인 작업.** 파일 삭제, 덮어쓰기, 데이터베이스
  구조 변경 같은 것들. 실수하면 복구가 안 되니까요.
- **의존성 추가, 비용 발생, 외부 서비스 연결.** 남의 라이브러리를 설치하거나
  돈이 들 수 있는 일은 주인의 허락이 필요해요.

**묻지 말고 진행해도 되는 경우 (단, "이렇게 정했어요"라고 밝히면서):**

- 프로젝트에 이미 있는 관례를 따르는 경우. 다른 파일들이 전부 어떤 스타일로
  돼 있으면 새 파일도 그렇게 하면 되죠. 물어볼 필요 없어요.
- 결과가 어차피 똑같은 내부 구현 선택.
- **업계 표준이 있는 경우.** 예를 들어 변경 이력 파일 이름은 전 세계 개발자가
  CHANGELOG.md라고 쓰기로 사실상 합의돼 있어요. 이런 건 표준을 따르고
  "표준이라서 이렇게 했어요"라고 한 줄 밝히면 됩니다.
- 파일 하나에만 영향 있고 바로 되돌릴 수 있는 사소한 선택.

전체를 요약한 한 줄이 매뉴얼에 박혀 있어요: **"되돌리기 어렵거나, 사용자 눈에
보이거나, 요구사항이 갈리면 → 묻는다. 아니면 관례를 따르고 선택을 밝힌다."**

그리고 예절 규칙 하나 — 질문은 3~4개까지 **모아서 한 번에** 물어봐야 해요.
5분마다 하나씩 물어보는 것도 사람을 지치게 하는 실패 방식이니까요.

### Part 2 — 숫자와 사실에는 근거를

AI의 고질병 중 하나가, 기억(학습 데이터)에 어렴풋이 남아 있는 숫자를 마치
공식 스펙처럼 자신 있게 말하는 거예요. "이 API는 하루 1만 번까지 호출
가능합니다"라고 말하는데, 확인해보면 문서에 그런 말이 없거나 옛날 얘기인 거죠.

그래서 규칙: 숫자·제한·가격·API 동작·버전 호환성을 말하려면 둘 중 하나를
반드시 해야 해요.

1. **실제 근거를 제시한다** — 공식 문서 주소, 코드의 정확한 위치(파일:줄번호),
   또는 방금 직접 실행해본 명령의 출력 결과.
2. **추정이라고 정직하게 라벨을 붙인다** — "추정입니다. 근거는 ○○예요."

그리고 우선순위가 정해져 있어요: **직접 실행해보는 게 1순위 > 공식 문서를
가져와서 읽는 게 2순위 > 그것도 안 되면 추정임을 밝히고 말하는 게 3순위.**
기억으로 대답하는 건 아예 선택지에 없어요.

마지막으로 "위험 신호" 목록이 있어요. AI가 스스로 멈춰야 하는 순간들:
"아마", "보통", "일반적으로 ~일 겁니다"라는 말로 숫자를 말하려는 순간,
문서를 안 열어보고 API 파라미터 이름을 쓰는 순간, 사용자가 고르지도 않은
옵션을 "당연히 이거겠지"라며 확정하는 순간.

## 3. write-gate — "만들기 전 검문소, 만든 후 검사소"

### 한 줄 요약
AI가 코드를 새로 만들기 **전**에는 "이미 있는 거 아니야?"를 확인시키고,
만든 **후**에는 10가지 품질 검사를 통과시키는 검문소 매뉴얼이에요.

### 왜 필요한가요?

AI의 가장 유명한 나쁜 버릇: 세션이 바뀌면 기억이 리셋되니까, 지난주에 만든
함수를 이번 주에 또 만들어요. 그래서 저장소에 거의 똑같은 함수가 3개, 4개
쌓여요. 나중에 버그를 고칠 때 하나만 고치면 나머지에서 같은 버그가 또 나오죠.
이걸 막으려면 "만들기 전에 검색부터"를 강제해야 해요.

### pre-write 모드 (코드 만들기 전) — 6단계 검문

**1단계: 존재 확인.** repo-xray 스캐너(아래 5번 스킬)를 실제로 돌려서,
만들려는 것과 비슷한 이름 2~3개를 검색해요. 중요한 건 AI의 기억이나 감이
아니라 **진짜 검색 프로그램을 실행**한다는 거예요. 예를 들어 "환율 변환
함수 만들어줘"라고 하면, exchange, convert, usd_to_krw 같은 이름들로
저장소 전체를 뒤져봐요.

**2단계: 외부 API 근거 확인.** 새로 쓰는 코드가 외부 라이브러리나
프레임워크의 API를 사용한다면 — 함수 이름, 파라미터, 옵션, 반환 형태 같은
버전에 민감한 부분 — AI의 기억으로 쓰면 안 돼요. AI의 기억은 학습 시점에
멈춰 있어서, 그 사이 라이브러리가 바뀌었으면 옛날 방식의 코드가 나오거든요.
그래서 context7이라는 최신 문서 조회 도구(MCP)가 연결되어 있으면 그걸로
공식 문서를 먼저 확인하고, 없으면 공식 문서를 직접 가져와 읽고, 그것도
안 되면 "추정입니다"라고 밝히고 써요. grounded-answers의 "기억으로 대답
금지" 원칙이 코드 작성에도 적용되는 거예요. 우리 프로젝트 내부 코드만
다루는 작업이면 이 단계는 건너뛰어요.

**3단계: 위치 확인.** 코드를 넣으려는 폴더의 MODULE.md(폴더 설명서)를 읽고,
"이 코드가 이 폴더의 담당 업무에 맞나?"를 확인해요. 주방에 세탁기를 놓으면
안 되니까요. 안 맞으면 맞는 폴더를 찾거나 사용자에게 물어봐요.

**4단계: 한 곳에만 두기(SSOT).** SSOT는 Single Source of Truth, "진실의
원천은 하나"라는 뜻이에요. 여러 곳에서 같이 쓰는 도구 함수나 타입은 정해진
공용 위치 한 곳에만 있어야 해요. 필요할 때마다 옆에 복사본을 만들면, 나중에
하나만 고치고 나머지를 까먹는 사고가 나요.

**5단계: 제자리에 배치.** 새 파일은 폴더 위계(어느 영역, 어느 층)에 맞게
만들어요. 저장소 맨 바닥에 파일을 수북이 쌓아두는 것(평탄화) 금지. 그리고
처음부터 파일을 나눠서 만들어요 — 한 파일에 다 몰아넣지 않고요.

**6단계: 경계 선언.** 코드를 쓰기 전에 "이 코드는 무엇을 가져다 쓸 수 있고,
누가 이 코드를 가져다 쓸 수 있는지"를 한 줄로 선언해요. 이걸 미리 정하면
나중에 스파게티처럼 얽히는 걸 예방할 수 있어요.

검문이 끝나면 판정은 반드시 셋 중 하나예요: **재사용**(있던 거 그대로 쓴다) /
**확장**(있던 걸 조금 고쳐 쓴다) / **신규**(진짜 없어서 새로 만든다 — 단,
"어디어디를 검색했는데 없었다"는 근거를 붙여서).

### post-write 모드 (코드 만든 후) — 10항목 검사

코드를 다 만들면 아래 10개를 하나씩 검사하고, 항목마다 통과(✅)/문제(⚠️)를
보고해요. 문제가 있으면 이유와 고치는 방법도 함께요. 단, 비례 원칙이 있어요 —
오타 수정 같은 작은 변경이면 무관한 항목은 "해당 없음" 한 마디로 넘어가요.
형식적으로 도배하는 것도, 귀찮다고 통째로 건너뛰는 것도 둘 다 실패거든요.
(9·10번 필수 항목만은 항상 답해요.)

1. **에러 삼킴 검사** — 이번에 만든 코드에 root-cause-first가 금지한 패턴
   (빈 except 등)이 들어갔는지. (v0.2.0부터는 훅이 코드를 쓰는 순간에도
   기계적으로 한 번 더 잡아요 — 7장 ⓪번 참고.)
2. **중첩 깊이 검사** — if 안에 if 안에 if... 3단계를 넘으면 읽기 힘들어져요.
   넘었으면 "일찍 돌아가기(early return)" 방식으로 펴야 해요.
3. **크기 검사** — 함수는 약 50줄, 파일은 약 400줄이 상한선. 넘으면 쪼개기를
   제안해요. (갓펑션/갓파일 방지)
4. **중복 vs 교체 검사** — 기존 코드를 대체하는 작업이었다면: 옛날 버전을
   지웠나? 옛날 것을 쓰던 곳들을 새것으로 연결했나? 주석으로 가려둔 시체
   코드가 남아 있지 않나?
5. **잔재 검사** — 이제 안 쓰는 import나 re-export(남의 것을 다시 내보내는
   통로)가 굴러다니지 않는지.
6. **순환 의존 검사** — A가 B를 쓰고 B가 다시 A를 쓰는 뱅글뱅글 관계가 새로
   생기지 않았는지. 의심되면 repo-xray 스캔을 돌려요.
7. **숨은 결합 검사** — 눈에 안 보이는 끈으로 모듈들을 묶지 않았는지: 전역
   상태 공유("어딘가의 변수를 우리 둘 다 몰래 쓴다"), 초기화 순서 의존("A를
   B보다 먼저 실행해야만 돌아간다"), import만 해도 부작용이 실행되는 코드,
   문서에 없는 암묵적 약속(호출 순서·데이터 형태를 그냥 가정), 그리고 한
   모듈에 모든 게 몰려드는 쏠림(fan-in/fan-out 과밀). AI 생성 코드에 거의
   항상 숨어 있는 비가시적 결합이라, 항목으로 못박았어요.
8. **테스트 품질 검사** — 테스트가 겉모양(문자열)만 비교하지 않고 실제 동작을
   검증하는지, 가짜(mock)는 외부 경계에서만 쓰는지, 빈 값·없음(None)·0·경계값
   같은 엣지케이스를 먼저 다뤘는지, 테스트를 통과시키려고 실제 코드에 꼼수를
   넣지 않았는지.
9. **실행 검증 (필수!)** — 변경한 코드를 **실제로 실행해서 눈으로 동작을
   봤는지.** 테스트를 돌렸거나, 앱을 켜봤거나, 버그 수정이면 원래 문제가
   났던 상황을 재현해서 이제 안 나는 걸 확인했거나. 왜 필수냐면 — 자기가
   쓴 코드를 자기 눈으로 다시 읽는 건 검증이 아니거든요. 쓸 때 했던 착각을
   읽을 때도 똑같이 해요. 실행을 못 했다면 "실행 검증 안 됨"과 그 이유를
   정직하게 말해야 하고, "될 겁니다"는 금지예요.
10. **문서 동기화 검사 (필수!)** — 파일을 추가/이동/이름변경했거나 폴더의
    역할이 바뀌었다면, 그 폴더의 MODULE.md와 CLAUDE.md의 폴더 지도를 **같은
    턴에** 고쳤는지. 실질적인 변경이면 CHANGELOG에 기록했는지.

그리고 형식 규칙이 하나 있어요: 검사 보고의 **마지막 두 줄은 반드시** ①"실행
검증: (무엇을 어떻게 돌려봤나)" 또는 "실행 검증 안 됨 — (이유)", ②"문서
갱신함: (목록)" 또는 "문서 영향 없음 — (이유)"로 끝나야 해요. 검증과 문서
얘기를 빼먹는 것 자체가 불가능하게 만든 거예요. 코드는 바뀌었는데 문서가
옛날 얘기를 하고 있으면, 다음 세션의 AI가 그 낡은 문서를 믿고 잘못된 판단을
해요. 거짓말하는 문서는 없는 문서보다 나쁘거든요.

### --deep 모드: "남의 눈" 설계 리뷰 (v0.2.0 신규)

체크리스트에는 근본적인 한계가 하나 있어요 — **자기가 쓴 코드를 자기가
검사한다**는 거예요. 쓸 때 한 착각은 검사할 때도 똑같이 하거든요. 사람
개발자들이 서로 코드 리뷰를 해주는 이유가 정확히 이거예요.

그래서 `/vibe-check:post-write --deep`을 치면, **이 코드를 쓴 기억이 전혀
없는 깨끗한 AI(부하 직원)**를 불러와요. Claude Code에 내장된 기능이라 별도
결제나 API 키가 필요 없어요. 이 리뷰어는 일부러 설계 이유를 전달받지 않고
— 변명을 들으면 같은 착각에 물들 수 있으니까 — 요구사항과 바뀐 파일만 보고
딱 5가지만 판단해요: 과잉 설계인가, 요청 안 한 게 끼었나, 더 단순한 길이
있었나, 숨은 결합을 만들었나, 다음 변경의 발목을 잡나. 스타일이나 버그
찾기는 안 해요 — 그건 체크리스트와 기계 게이트 몫이고, 이 눈은 **판단
착오**만 잡아요. 근거(파일:줄번호) 없는 지적은 금지고, 판정은 "통과" 또는
"재고 권장 N건"으로 딱 떨어져요. 큰 변경(새 기능, 새 파일, 리팩토링) 후에
쓰세요 — 오타 수정에 부를 필요는 없어요.

## 4. docs-keeper — "문서 4종을 관리하는 사서"

### 한 줄 요약
프로젝트의 기억을 담당하는 문서 4개(CLAUDE.md, MODULE.md, handover.md,
CHANGELOG.md)를 만들고, 형식을 지키고, 계속 신선하게 유지하는 사서 매뉴얼이에요.

### 문서 4개는 각각 무슨 일을 하나요?

핵심 원칙은 **"문서마다 직업이 딱 하나"**예요. 역할이 섞이면 아무도 안 읽는
잡동사니가 되니까요.

**CLAUDE.md — 프로젝트의 지도.** AI가 세션을 시작할 때마다 자동으로 읽는
파일이에요. 프로젝트 개요(3~5줄), 꼭 지켜야 할 요구사항, 실행 방법, 그리고
"어느 폴더가 무슨 일을 하는지" 폴더 지도가 들어가요. 여기서 제일 중요한 규칙:
**120줄을 넘기면 안 돼요.** 왜냐하면 이 파일은 매 세션 통째로 AI 머릿속에
들어가서, 한 줄 한 줄이 영원히 반복되는 비용이거든요. 그래서 지도는 얇게,
자세한 건 아래층(MODULE.md)으로 내려보내요.

**MODULE.md — 각 폴더의 상세 설명서.** 기능을 담당하는 폴더마다 하나씩 둬요.
"이 폴더가 하는 일(그리고 안 하는 일)", 주요 파일 목록, 이 기능에 필요한
데이터 모델과 설계, 이 폴더가 의존하는 것과 이 폴더를 쓰는 곳, 과거에 깨졌던
함정. 목표는 — **새 세션의 AI가 이 문서만 읽고도 이 폴더에서 바로 일을 시작할
수 있을 만큼.**

**handover.md — 세션 인수인계 일지.** AI의 기억은 세션이 끝나거나 압축(compact)
되면 날아가요. 그래서 병원의 간호사 교대 일지처럼, "이번 근무(세션)에 무슨 일이
있었고, 뭐가 진행 중이고, 다음 사람이 뭘 조심해야 하는지"를 남기는 파일이에요.
날짜와 시간이 붙은 항목이 최신순으로 쌓여요. 각 항목에는 4가지가 들어가요:
한 일 / 진행 중 / 다음에 할 일 / 주의·함정. 그리고 쓰는 요령이 명시돼 있어요 —
일기가 아니라 **미래의 나에게 보내는 업무 지시**처럼 쓰라고요. "왜 그렇게
결정했는지", "이미 해봤는데 안 됐던 방법"까지 적어야 다음 세션이 같은 삽질을
반복하지 않아요.

**CHANGELOG.md — 변경 이력 장부.** handover가 "맥락"이라면 이건 "결과"예요.
무엇이 언제 바뀌었는지만 담백하게 기록해요. 전 세계 표준인 Keep a Changelog
형식을 따라서 Added(추가)/Changed(변경)/Fixed(수정)/Removed(제거)로 분류하고,
각 항목에 날짜·시간을 붙여요. 중요한 필터: **실질적인 변경만** 기록해요.
동작·API·데이터 구조·의존성·설정이 바뀐 것만요. 오타 수정이나 코드 정리처럼
동작이 안 바뀐 건 기록하지 않아요 — 장부가 소음으로 가득 차면 아무도 안 읽으니까.

### 사서의 업무 모드 5가지

**init 모드** (`/vibe-check:init`) — 프로젝트에 문서 시스템을 처음 설치해요.
뭐가 이미 있고 뭐가 없는지 확인하고, 없는 것만 템플릿으로 만들어요. 이미 있는
파일은 **절대 조용히 덮어쓰지 않고** 먼저 물어봐요. 폴더들을 훑어서 폴더 지도
초안을 만들고, "어느 폴더에 MODULE.md를 만들까요?"도 물어봐요. 그리고 이 설치가
끝나야 훅(자동 기능)들이 이 프로젝트에서 켜져요.

**handover 모드** (`/vibe-check:handover`) — 인수인계 항목을 하나 써서 맨 위에
추가해요. 시간은 컴퓨터의 date 명령을 실제로 실행해서 받아와요 — AI가 시간을
추측(환각)하면 안 되니까요. 옛 항목은 절대 수정하지 않고, 항목이 20개를 넘으면
오래된 절반을 handover-archive.md라는 창고 파일로 옮겨요(파일이 무한정 커지는
것 방지).

**log 모드** (`/vibe-check:log`) — 먼저 git(버전 관리 도구)의 기록을 확인해서
실제로 뭐가 바뀌었는지 파악하고, 사소한 것을 걸러낸 뒤, 실질 변경만 CHANGELOG에
분류해 넣어요. 기록할 게 없으면 억지로 채우지 않고 "없다"고 말해요.

**recall 모드** (`/vibe-check:recall`, v0.2.0 신규) — 사서가 기록을 쓰기만
하는 게 아니라 **찾아주기도** 해요. "예전에 시세 API 왜 폴링으로 바꿨지?"
라고 물으면(명령어 없이 그냥 물어도 돼요), 인수인계 공책과 창고
(handover-archive), CHANGELOG를 전부 뒤져서 — 검색어도 한국어/영어/코드
이름으로 바꿔가며 여러 번 — 걸린 기록을 읽고 **"2026-05-12 기록에 따르면
…"처럼 날짜와 출처를 붙여서** 답해요. 못 찾으면 "어디를 무슨 단어로
뒤졌는데 없었다"고 정직하게 말하고요. 한 가지 기억할 것: 이건 **적어둔
것만** 찾을 수 있어요. 그래서 인수인계를 잘 남기는 습관이 곧 나중에 꺼낼
수 있는 기억의 양이 돼요.

**welcome 모드** (`/vibe-check:welcome`) — 처음 쓰는 사람에게 문서 4종과 명령어
지도를 친절한 한국어로, 짧게 안내해요.

## 5. repo-xray — "저장소 엑스레이 촬영 기사"

### 한 줄 요약
저장소 전체를 스캔해서 "중복 함수, 안 쓰이는 것 같은 코드, 너무 큰 파일"의
**증거(JSON 파일)**를 뽑아내는 실제 프로그램 + 그 증거를 정직하게 읽는 법을
가르치는 매뉴얼이에요. (지현님이 이전에 만들어뒀던 걸 플러그인에 편입했어요.)

### 왜 "증거"라는 말을 쓰나요?

AI에게 "우리 저장소에 중복 코드 있어?"라고 물으면, AI는 파일 몇 개를 대충
훑어보고 "없는 것 같아요"라고 **추측**으로 대답하는 경향이 있어요. 사람으로
치면 의사가 엑스레이도 안 찍고 "뼈는 괜찮아 보여요"라고 말하는 거죠.

repo-xray는 그 엑스레이 기계예요. 진짜 스캔 프로그램(파이썬 한 파일, 별도 설치
필요 없음)이 저장소의 모든 코드 파일을 읽고 분석해서 결과를 JSON 파일로 남겨요.
읽을 수 있는 코드는 파이썬과 자바스크립트 계열 전부 — v0.2.0부터 요즘 웹
프로젝트의 표준인 **타입스크립트(`.ts`)와 리액트 파일(`.tsx`/`.jsx`)**까지
포함이에요. 그리고 매뉴얼은 AI에게 이렇게 강제해요:

- **스캔을 안 돌렸으면 개수를 말하지 마라.** ("중복이 12개 있어요"라고 말하려면
  방금 스캔을 돌렸어야 함)
- **"없다"고 말할 때는 반드시 어디를 찾아봤는지(스캔 범위)를 함께 말해라.**
  ("없어요"가 아니라 "백엔드·프론트 121개 파일을 스캔했는데 없었어요")

### 기능 1: scan (전체 촬영)

저장소 전체를 훑어서 이런 것들을 찾아요.

- **중복 함수**: 이름이 달라도 잡아요. 코드를 문장이 아니라 문법 구조(AST,
  추상 구문 나무)로 바꿔서 비교하는데, 함수 이름뿐 아니라 **안에서 쓰는 변수
  이름까지 지운 뒤** 비교해요. 그래서 이름만 싹 바꾼 복사본도 "구조가 같은
  쌍둥이"로 걸려요. 이건 아주 강한 증거라서 자신 있게 "진짜 중복"이라고
  말해도 돼요.
- **유사 중복 (v0.2.0 신규)**: 완전히 같지는 않고 **90% 이상 비슷한** 함수
  쌍도 따로 찾아내요. 이게 왜 중요하냐면 — AI가 실제로 저지르는 중복은 복사·
  붙여넣기가 아니라 "지난번 것과 거의 같은데 한두 줄만 다른 재구현"이거든요.
  완전 일치만 찾으면 정확히 그 실수를 놓쳐요. 단, 이건 중복 "확정"이 아니라
  "의심"이라서, AI는 두 함수를 다 읽어본 뒤에만 합치자고 제안할 수 있어요.
- **미사용 후보**: 정의는 돼 있는데 저장소 어디에서도 불려지는 흔적이 없는
  함수들. 단, 이름이 "후보"인 이유가 중요해요 — 아래에서 설명할게요.
- **너무 큰 파일**: 갓파일이 되어가는 파일들.

### 기능 2: find (특정 부위 촬영)

"이런 함수 이미 있어?"에 답하는 검색이에요. 정확히 같은 이름뿐 아니라 **비슷한
이름**까지 찾아줘요. 사용자가 이름이 아니라 "환율 바꾸는 거"처럼 기능으로
설명하면, AI가 그럴듯한 이름 2~3개(exchange, usd_to_krw, convertRate)로 여러 번
검색해보게 돼 있어요.

### 정직함 장치 — 이 스킬의 백미

**"미사용 후보"는 판결이 아니에요.** 스캐너는 코드에 적힌 이름만 셀 수 있어요.
그런데 프로그램 세계에는 이름을 문자열로 조립해서 부르는 경우(동적 호출)가
있어서, 스캐너 눈에는 안 쓰이는 것처럼 보여도 실제로는 쓰이는 코드가 있어요.
그래서 AI는 "이거 지워도 돼요"라고 단정하지 못하고, "N개 파일을 스캔했는데
참조를 못 찾았어요 — 지우기 전에 ○○만 한번 확인해보세요"라고 말해야 해요.

특히 중요한 예외: **데코레이터가 붙은 파이썬 함수는 삭제 제안 금지.** 지현님의
FastAPI 백엔드에서 `@app.get("/price")` 같은 게 붙은 함수들은, 코드 어디에서도
이름으로 불리지 않아요. 웹 프레임워크가 "등록"이라는 방식으로 몰래 부르거든요.
스캐너 눈에는 미사용처럼 보이지만 지우면 서비스가 죽어요. 그래서 이런 함수는
스캐너 결과에 "decorated(데코레이터 있음)" 표시가 붙고, AI는 삭제를 제안할 수
없게 돼 있어요. 같은 이유로 **테스트 파일(`test_*.py` 등) 안의 심볼도 미사용
후보에서 빼요** — 테스트 실행기가 이름 규칙으로 알아서 부르니까, 아무도 참조
안 해도 살아 있는 코드거든요. (v0.2.0에서 플러그인이 자기 저장소를 자기
스캐너로 찍어보다가 발견해서 고친 오탐이에요.)

그리고 이런 "속는 방식"들을 **오탐 인덱스**라는 문서에 계속 쌓아요
(v0.2.0 신규). 프레임워크 등록, 동적 호출, 스캔 안 되는 파일 종류, 공개
API 같은 알려진 함정 9종과 각각 어떻게 말해야 하는지가 적혀 있고, AI는
스캔 결과를 보여주기 전에 반드시 이 목록과 대조해요. 새 오탐이 확인되면
목록에 추가하고 재발 방지 테스트까지 만드는 게 규칙이에요 — 오래 두들겨
맞으며 정확해지는 구조죠.

**문서에 적힌 이름은 "사용"이 아니에요 (v0.2.0 수정).** 예전엔 MODULE.md나
CHANGELOG에 함수 이름이 적혀 있으면 스캐너가 "어딘가에서 쓰이는구나"로
착각했어요. 그러면 이 플러그인이 시키는 대로 문서를 열심히 쓸수록 죽은 코드를
못 찾게 되는 모순이 생겨요. 이제 문서 속 언급은 따로 세서 `doc_mentions`라는
칸에 담아요 — "이 코드를 지우면 이 문서들도 같이 고치세요" 목록으로 쓰라고요.

### 답변 태도

증거는 차갑게, 설명은 따뜻하게. 결과를 JSON 덩어리로 던지지 않고, 일상 언어로
핵심만(한 번에 최대 3가지 제안), 각 제안에 파일:줄번호 증거를 붙여서 말해요.
전체 목록이 필요하면 "다 보여드릴까요?"라고 물어보고요.

## 6. guards-setup — "표지판을 과속방지턱으로 바꾸는 시공사"

### 한 줄 요약
지금까지의 스킬들이 AI에게 말로 가르치는 "표지판"이라면, 이 스킬은 규칙을
어긴 코드를 기계가 물리적으로 거부하게 만드는 "과속방지턱"을 시공해요.

### 왜 필요한가요?

솔직한 사실 하나: AI는 말로 가르친 규칙을 **확률적으로만** 지켜요. 대화가
길어지면 잊어버리기도 해요. 그런데 린트(lint)라는 검사 도구가 "이 함수는
복잡도가 기준을 넘었으니 불합격"이라고 빨간불을 켜면, 그건 잊을 수가 없어요.
AI도 그 빨간불을 보고 고칠 수밖에 없고요. 그래서 지현님 체크리스트의 "4.
강제해야 할 것들"이 이 스킬이 된 거예요. 규율의 절반은 애초에 AI에게 시킬 게
아니라 기계에게 시킬 일이에요.

### 설치해주는 방지턱들

**파이썬 프로젝트용:**
- **ruff(러프)**: 코드 검사기. 함수가 너무 복잡하면(분기 10개 초과), 인자가
  너무 많으면(5개 초과), 안 쓰는 import가 굴러다니면 경고를 울려요.
  갓펑션·과중첩을 기계가 거부하는 거예요.
- **mypy(마이파이)**: 타입 검사기. "이 함수는 숫자를 받아야 하는데 글자를
  넣었네요" 같은 오류를 실행 전에 잡아요. 코드에 숨어 있는 암묵적 약속을
  겉으로 드러내는 도구예요. 입문자에게는 처음부터 최고 강도(strict)가 아니라
  1단계 → 2단계 점진 적용을 제안하게 돼 있어요.
- **import-linter(임포트 린터)**: 건물 층간 질서 감독관. "routers(위층)는
  core(아래층)를 쓸 수 있지만, core는 routers를 알면 안 된다" 같은 규칙을
  설정해두면, 위반하는 순간 검사가 실패해요. 순환 의존(A↔B 뱅글뱅글)도 여기서
  막아요. 층 구조 초안은 CLAUDE.md의 폴더 지도를 보고 만들어서 사용자와 확인해요.

**자바스크립트/타입스크립트 프로젝트용:**
- **eslint(이에스린트)**: ruff의 JS판. 중첩 3단계 초과, 60줄 넘는 함수, 인자
  5개 이상, 빈 catch를 거부해요.
- **dpdm**: 순환 의존 탐지기. 발견되면 검사 실패(exit 1)로 처리해요.

**CI 워크플로 (GitHub에 올리는 자동 검사):**
- **vibe-guards.yml**: 코드를 GitHub에 올릴 때마다(push/PR) 위의 검사를 전부
  자동 실행하고, 하나라도 실패하면 빨간 X를 띄워요.
- **biweekly-audit.yml**: 매월 1일과 15일에, GitHub의 컴퓨터에서 Claude가
  자동으로 저장소 구조를 감사하고 보고서를 이슈로 올려요. 낭비 방지 장치도
  있어요 — 지난 감사 이후 코드 변경이 없으면(문서만 바뀐 건 제외) 그냥 건너뛰어요.

### 시공 절차 — 순서가 생명

1. **감지**: 프로젝트가 파이썬인지 JS인지 (둘 다인지) 파일들을 보고 판단.
2. **기존 설정 읽기**: 이미 린트 설정이 있으면 먼저 읽어요. 절대 덮어쓰지 않아요.
3. **묻기 (필수)**: "이 중에 뭘 설치할까요?"를 선택지로 물어봐요. 항목마다
   효과 설명 한 줄씩 붙여서. 제멋대로 설치 금지. (grounded-answers 규율이
   여기서도 작동하죠.)
4. **병합**: 새 설정을 기존 설정에 끼워 넣되, 사용자가 이미 정해둔 값은 건드리지
   않아요.
5. **실측 검증**: 설치가 끝나면 도구를 **실제로 한 번 돌려서 진짜 출력**을
   보여줘요. "잘 될 겁니다"는 보고가 아니에요.

그리고 중요한 금지 조항 — 검사를 돌렸더니 위반이 500개 쏟아졌다고 해서 기준을
느슨하게 풀어서 통과시키는 것, **금지**예요. 그건 시험이 어렵다고 커트라인을
내리는 거고, root-cause-first 위반이에요. 대신 "기존 코드는 경고만, 새 코드부터
에러" 같은 점진 적용을 제안하게 돼 있어요.

## 7. 보너스: 스킬이 아닌 "훅" 4개 — 완전 자동으로 도는 것들

스킬은 AI가 상황을 인식해야 발동하지만, 훅(hook)은 **AI와 상관없이 특정 사건이
일어나면 기계적으로 실행되는 장치**예요. vibe-check에는 4개가 있어요.

**⓪ 에러 삼킴 + 비밀키 감시 (PostToolUse 훅, v0.2.0 신규).** AI가 코드를 쓰는
순간마다 — AI의 양심이 아니라 **정규식이** — 두 가지를 검사해요. 첫째, "에러를
조용히 무시하는 패턴"(빈 except/catch)이 새로 생겼는지. 발견되면 "원인부터
찾으라"는 규율을 즉시 되새겨줘요. 둘째, **API 키(유료 서비스 비밀번호)가 코드에
그대로 적혔는지.** 이게 왜 중요하냐면 — 초보가 제일 많이 내는 보안 사고가
해킹이 아니라, 키를 코드에 적은 채 GitHub에 올리는 거예요. 공개 저장소의 키는
봇이 몇 분 안에 주워가서 요금 폭탄으로 돌아와요. 감지되면 "키를 .env 파일로
옮기고, 이미 올렸다면 키를 재발급하라"고 그 자리에서 안내해요. (.env 파일
자체는 키의 올바른 집이라 검사 안 해요.) root-cause-first 매뉴얼(1장)이
"표지판"이라면 이 훅은 "과속 카메라"예요 — AI가 매뉴얼을 안 읽었어도 걸려요.
정말 의도한 경우라면 그 줄에 `vibe-check: allow-swallow`(에러 삼킴) /
`vibe-check: allow-secret`(테스트용 가짜 키) 주석을 달면 통과예요.

**① compact 직전 자동 인수인계 (PreCompact 훅).** AI의 대화 기억이 꽉 차면
Claude Code가 기억을 압축(compact)하는데, 이때 세부 내용이 뭉개져요. 이 훅은
압축이 시작되기 직전에 — AI가 아니라 **파이썬 스크립트가 직접** — 대화 기록
파일을 읽어서 "최근 사용자 요청 5개 + 수정한 파일 목록"을 날짜·시간과 함께
handover.md에 기록해요. AI에게 시키면 까먹을 수 있지만 스크립트는 못 까먹어요.
100% 보장이라는 뜻이에요.

**② 세션 시작 주입 (SessionStart 훅).** 새 세션이 시작되거나 압축 직후에,
최근 인수인계 내용과 규율 요약 한 줄("에러는 원인부터, 근거 없는 단정 금지,
만들기 전에 검색")을 AI 머릿속에 자동으로 넣어줘요. 압축 직후라면 "방금 자동
기록된 인수인계를 읽고 더 좋게 다듬어라"는 지시도 함께요.

**③ 기록 깜빡임 알림 (Stop 훅).** 세션이 끝날 때, 코드는 바꿨는데 CHANGELOG
기록을 안 했으면 "기록할까요?"라고 알려줘요. 잔소리가 되지 않게 **세션당 딱
한 번만** 알리고, vibe-check를 설치(init)한 프로젝트에서만 작동해요.

**훅이 진짜 돌고 있는지 확인하는 법 — `/vibe-check:doctor` (v0.2.0 신규).**
훅에는 설계상의 딜레마가 하나 있어요. 훅이 고장 났을 때 Claude Code까지 같이
멈추면 안 되니까, 모든 훅은 "무슨 일이 있어도 조용히 넘어가게" 만들어져 있어요.
그런데 그 대가로, 예를 들어 컴퓨터에 python3가 없으면 **안전벨트 4개가 전부
티도 안 나게 꺼진 상태**가 될 수 있어요. 꺼진 줄 모르고 믿는 게 최악이죠.
그래서 doctor는 훅 4개와 스캐너를 **말이 아니라 실제로 실행해서** ✅/❌로
보여줘요 — 임시 프로젝트를 하나 만들어 각 훅에 가짜 상황을 먹여보고, 진짜로
handover가 기록되는지, 에러 삼킴이 감지되는지까지 끝단 확인을 해요. 플러그인을
설치한 직후에 한 번 꼭 돌려보세요.

## 8. 전체 그림 — 이 여섯 권이 하나로 노리는 것

여섯 스킬은 따로따로가 아니라 하나의 척추를 공유해요: **"증거 없이 주장하지
않는다."**

- repo-xray는 그 원칙을 **저장소 내부 사실**에 적용하고 (스캔 없이 "중복 없다"
  금지),
- grounded-answers는 **바깥 세상 사실**에 적용하고 (문서/실측 없이 "API 제한은
  1만" 금지),
- guards-setup은 **자기 작업 결과**에 적용하고 (실행해보지 않고 "설치 완료"
  금지),
- root-cause-first는 **버그의 원인**에 적용하고 (원인 규명 없이 "고쳤다" 금지),
- write-gate는 **"새로 만들어야 한다"는 판단**에 적용하고 (검색 없이 "없으니까
  새로 만들자" 금지),
- docs-keeper는 그 모든 결과가 **다음 세션까지 살아남게** 만들어요 (기록 없이
  기억을 신뢰하는 것 금지).

바이브코딩의 문제들 — 중복 재구현, 땜빵, 임의 결정, 숫자 환각, 갓파일, 문서
거짓말 — 은 결국 전부 "확인하지 않고 자신 있게 진행하는 것"에서 나와요.
vibe-check는 그 한 가지 뿌리를 여섯 방향에서 막는 플러그인이에요.

## 9. v0.2.0에서 달라진 것 (요약)

- **과속 카메라 추가**: 에러 삼킴과 **하드코딩된 API 키**를 훅이 기계적으로
  감지해요 (7장 ⓪번). 규율의 절반이 "말"에서 "기계"로 내려갔어요.
- **"남의 눈" 설계 리뷰** (`post-write --deep`): 코드를 쓴 기억이 없는
  깨끗한 AI를 불러 과잉 설계·스코프 크립 같은 판단 착오를 검토 (3장 끝).
  별도 비용 없이 Claude Code 안에서 돌아가요.
- **기억 검색** (`/vibe-check:recall`): "예전에 왜 이렇게 했지?"에 기억이
  아니라 기록으로 — 공책·창고·CHANGELOG를 뒤져 날짜·출처와 함께 답해요
  (4장 recall 모드).
- **`/vibe-check:doctor`**: 훅과 스캐너를 실제로 돌려보고 ✅/❌로 보여주는
  자가진단. 훅은 고장 나도 조용히 넘어가게 설계돼 있어서(Claude Code를 절대
  방해하지 않기 위해), "꺼진 줄 모르고 믿는 상태"를 막으려면 이게 필요해요.
  설치 직후 한 번 꼭 돌려보세요.
- **repo-xray가 TypeScript를 읽어요**: `.ts`/`.tsx`/`.jsx`까지 스캔. React·
  Next.js 프로젝트에서도 "이 함수 이미 있어?"가 작동해요.
- **"90% 비슷한 재구현"도 잡아요**: 똑같은 복붙만이 아니라, 변수 이름만 바꾸거나
  한두 줄만 다른 함수 쌍도 찾아내요 — AI가 실제로 저지르는 중복은 이쪽이거든요.
- **문서 언급과 코드 사용을 구분해요**: 예전엔 MODULE.md에 함수 이름이 적혀
  있으면 "사용 중"으로 착각했는데, 이제 문서 언급은 따로 세요. 문서를 열심히
  쓸수록 죽은 코드를 못 찾던 모순이 고쳐졌어요.
- **검사소(post-write)에 두 항목 추가**: "숨은 결합을 만들지 않았나"와
  **"실제로 실행해서 확인했나"**. 특히 후자 — 자기가 쓴 코드를 자기 눈으로
  읽는 건 검증이 아니에요. 실행을 못 했으면 "못 했다"고 말하게 해요.
- **플러그인 자신도 검사받아요**: 스캐너와 훅에 테스트 34개 + GitHub Actions
  CI. 규율 파는 도구가 자기 규율을 먹기 시작했어요.
