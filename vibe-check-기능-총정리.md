# vibe-check 기능 총정리 (v0.2.0)

> 한 장으로 보는 "내 플러그인에 뭐가 들어 있나".
> 자세한 원리는 [스킬 설명서](./vibe-check-스킬-설명서.md), 설계 이유는
> [평가·개선 기록](./vibe-check-평가-및-개선-기록.md) 참고.

vibe-check는 3겹으로 일해요:

1. **자동 (훅)** — 내가 아무것도 안 해도 기계가 알아서
2. **명령어** — 내가 필요할 때 부르는 것
3. **규율 (스킬)** — AI가 상황을 인식하면 스스로 펼쳐 읽는 매뉴얼

---

## 1겹. 자동으로 도는 것 (훅 4개)

| 언제 | 뭘 해주나 |
|---|---|
| AI가 코드를 쓴 직후 | **에러 삼킴**(빈 except/catch)과 **하드코딩된 API 키**가 새로 생기면 즉시 감지 → AI에게 "원인부터 고쳐라 / 키는 .env로 옮기고 이미 푸시했으면 재발급하라"고 그 자리에서 지시 |
| 기억 압축(compact) 직전 | 이번 세션의 요청·수정 파일을 **스크립트가 직접** handover.md에 기록 — AI가 까먹어도 기록은 남음 |
| 세션 시작/압축 직후 | 최근 인수인계 + 규율 요약 한 줄을 AI 머릿속에 자동 주입 |
| 세션 종료 시 | 코드는 바꿨는데 CHANGELOG를 안 썼으면 알림 (세션당 1번만) |

- 훅은 `init` 한 프로젝트에서만 켜지고, **절대 작업을 막지 않아요** (경고만).
- 의도된 예외는 그 줄에 주석: `vibe-check: allow-swallow` (에러 삼킴) /
  `vibe-check: allow-secret` (테스트용 가짜 키)

## 2겹. 명령어 10개

**설치할 때 (프로젝트마다 1회)**

| 명령 | 뭐 하는 것 |
|---|---|
| `/vibe-check:welcome` | 뭐부터 할지 모를 때 첫 안내 |
| `/vibe-check:doctor` | 훅·스캐너를 **실제로 돌려서** 작동 확인 (✅/❌) — 설치 직후 필수 1회 |
| `/vibe-check:init` | 문서 4종 설치 + 훅 활성화 |

**평소 코딩할 때**

| 명령 | 뭐 하는 것 |
|---|---|
| `/vibe-check:pre-write 만들려는 것` | 만들기 **전**: 이미 있는지 스캐너로 검색 → 재사용/확장/신규 판정 |
| `/vibe-check:post-write` | 만든 **후**: 10항목 체크리스트 (에러 삼킴, 크기, 중복, 숨은 결합, **실행 검증 필수**, 문서 동기화 필수) |
| `/vibe-check:post-write --deep` | 큰 변경이면: **"남의 눈"** — 이 코드를 쓴 기억이 없는 깨끗한 AI를 불러 과잉 설계·불필요한 기능·더 단순한 길을 검토 |

**기록할 때 / 찾을 때**

| 명령 | 뭐 하는 것 |
|---|---|
| `/vibe-check:handover` | 세션 마칠 때 인수인계 남기기 (다음 세션의 나에게) |
| `/vibe-check:log` | 실질 변경을 CHANGELOG에 기록 (사소한 건 거름) |
| `/vibe-check:recall 궁금한 것` | **기억 검색** — "예전에 왜 이렇게 했지?"를 기록에서 찾아 날짜·출처와 함께 답변 |

**가끔**

| 명령 | 뭐 하는 것 |
|---|---|
| `/vibe-check:audit` | 저장소 전체 스캔: 중복(변수명 달라도), **90% 유사 중복**, 미사용 후보, 갓파일 — Python + JS/TS(.ts/.tsx 포함) |
| `/vibe-check:guards` | 린트/타입/순환의존 검사를 기계로 강제 설치 (`--ci`면 GitHub 자동 검사까지) |

## 3겹. AI가 알아서 지키는 규율 (스킬 6개)

명령어 없이도, 대화 내용이 트리거에 걸리면 AI가 스스로 매뉴얼을 읽어요:

| 스킬 | 언제 발동 | 핵심 규칙 |
|---|---|---|
| root-cause-first | 에러/버그/fallback 얘기 | 증상 가리기 금지, 원인부터. fallback은 3조건 충족 시에만 |
| grounded-answers | 애매한 결정, 숫자/스펙 질문 | 되돌리기 어려우면 묻기. 숫자는 근거 제시 or "추정" 라벨 |
| write-gate | 만들어줘/추가해줘/리뷰해줘 | 위 pre-write/post-write 게이트 |
| docs-keeper | 문서/인수인계/기록/과거 질문 | 문서 4종 관리 + recall. CLAUDE.md는 120줄 이하 유지 |
| repo-xray | 중복 있어? 이 함수 있어? | **스캔 없이 주장 금지**, 부재 주장엔 스캔 범위 명시, 오탐 인덱스 대조 |
| guards-setup | 린트/CI 세팅 요청 | 묻고 설치, 기존 설정 보존, 설치 후 실측 검증 |

## 프로젝트에 생기는 문서 4종

| 문서 | 역할 |
|---|---|
| `CLAUDE.md` | 프로젝트 지도 (얇게 — 매 세션 로드되므로) |
| `폴더/MODULE.md` | 각 폴더의 상세 설계 |
| `handover.md` | 세션 인수인계 (20개 넘으면 자동 아카이브, recall로 검색 가능) |
| `CHANGELOG.md` | 실질 변경 이력 |

## 버전 한 줄 요약

- **v0.1.0** — 뼈대: 문서 4종 + 규율 스킬 + 스캐너 + 훅 3개
- **v0.2.0** — 기계 층 강화: 에러삼킴·비밀키 감지 훅, 남의 눈 리뷰(--deep),
  기억 검색(recall), doctor 자가진단, 스캐너 TS 지원·유사중복·오탐 인덱스,
  테스트 43개 + CI

## 철학 한 줄

> **확인하지 않고 자신 있게 진행하는 것** — 바이브코딩 사고의 뿌리는 이
> 하나고, vibe-check는 그걸 기억(문서)·규율(스킬)·강제(훅/린트) 세 방향에서
> 막는다. 그리고 규율은 말보다 기계에 맡길수록 안 잊힌다.
