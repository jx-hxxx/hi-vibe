이런 실수, 막아줘요

AI에게 코딩을 맡기면 이런 일이 자주 생겨요.

빠르게 만들 수 있지만, 프로젝트가 커질수록 작은 실수가 조용히 쌓여요.

있던 기능을 또 만들어요이전에 만든 코드를 잊고 비슷한 코드를 새로 만들어요. (예: 같은 함수·헬퍼를 또)
오류를 고친 척하고 숨겨요원인을 해결하지 않고 임시방편으로 넘어가 나중에 더 크게 터져요. (예: fallback·기본값으로 덮기)
확인하지 않고 마음대로 정해요애매한 요구사항이나 최신 정보를 확인하지 않고 추측해요. (예: API 제한·가격·버전)
코드가 점점 손대기 어려워져요한곳에 기능을 계속 쌓아 나중에는 수정하기가 무서워져요. (예: 엣지케이스 누락·갓파일)

첫째 겹

기억을 대신 적어둬요

무슨 프로젝트인지, 어디까지 했는지, 뭐가 바뀌었는지를 자동으로 적어둬요. 그래서 대화가 새로 시작돼도 앞 내용을 까먹지 않아요.

둘째 겹

AI한테 규칙을 심어요

“만들기 전에 있는지 먼저 찾아봐”, “에러는 원인부터 봐”, “확인 안 한 건 단정하지 마” 같은 규칙을 AI한테 미리 심어서, 스스로 지키게 해요.

셋째 겹

기계가 직접 막아요

코드를 자동으로 검사하는 도구가 위험한 코드를 그 자리에서 잡아요. 원하면 GitHub에 올릴 때 아예 통과 못 하게 막을 수도 있어요.

기능 하나하나

설치하면, 프로젝트에 이런 게 생겨요.

추상적인 "좋아져요"가 아니라, 실제로 만들어지고 봐주는 것들이에요.
핵심은 문서를 한 파일에 몰아넣지 않고 여러 곳에 나눠 두고, 코드가 바뀌면 자동으로 따라 갱신한다는 거예요.

📁 내 프로젝트/ — /hi-vibe:init은 가볍게 시작하고, 나머지 문서는 필요할 때 알아서 생겨요
  • CLAUDE.md프로젝트 전체 지도 — 개요·요구사항·폴더 구조 (init 때)
  • handover.md세션 인수인계 — 다음 대화가 맥락 안 잃게. 대화 정리(compact) 때 자동 기록돼요 (init 때)
  • api/MODULE.md그 폴더의 상세 설계·데이터·주의점 — 폴더가 복잡해질 때 생겨요 (폴더마다 하나씩)
  • CHANGELOG.md변경 이력 — 뭐가 언제 왜 바뀌었나. 첫 /hi-vibe:log 때 생겨요
자동 예방⚡ AUTO
🔍중복 재구현 방지find

새 함수·파일을 만들기 전에 "이미 있는지" 먼저 찾아봐요. AI가 세션마다 있던 걸 또 만드는 걸 막죠. 없을 땐 "스캔한 N개 파일에서 못 찾았어요"라고 범위까지 밝혀요.

🩹에러 땜빵 금지즉시 감지

에러를 fallback으로 덮는 대신 원인부터 보게 해요. 몰래 삼키는 코드(빈 catch, except: pass)를 쓰는 순간 감지해요. 의도한 거면 주석 한 줄로 통과.

📊근거 없는 단정 금지규율

"아마 이럴 거예요"를 사실처럼 말하지 않게 해요. 수치·API는 확인하고 답하거나, 못 하면 "추정입니다"라고 밝혀요. (context7 연결 시 최신 문서까지 조회)

기억·검토⚡ AUTO
👀딴 클로드가 설계 재검토review --deep

코드를 짠 기억이 전혀 없는 딴 클로드를 불러와 방금 짠 걸 봐요 — 편견 없이. 과잉설계·불필요한 기능·숨은 결합을 "이건 좀 과한데?" 하고 짚어줘요.

🧠기억·인수인계·검색handover · recall

대화가 정리(compact)될 때마다 진행상황을 자동 기록하고, 새 대화엔 최근 인수인계를 다시 넣어줘요. "예전에 왜 이렇게 했지?" 하면 날짜·이유까지 답해요.

🎯오탐 방지규율·스캐너

"안 쓰는 코드"라고 잘못 짚는 걸 막아요. 프레임워크가 이름 없이 부르는 함수·테스트·export default·만드는 중인 코드를 죽은 코드로 오해하지 않게 인덱스로 관리해요.

직접 점검🖐 MANUAL
🔬구조 점검check

이미 코드가 많은 프로젝트를 스캔해 "90% 비슷하게 또 만든" 함수·안 쓰는 코드·너무 큰 파일까지 짚어줘요. 지우기 전 "아직 만드는 중"인지 구분해요.

🛡️기계 강제 (린트·CI)gate --ci

gate면 로컬에 검사기(ruff·eslint·타입·순환의존)를 설치해 빨간 줄로 표시. gate --ci면 GitHub push마다 검문 — 위반은 빌드 실패로 막아요.

🩺자가진단doctor

hook은 방해 안 하려고 조용히 실패하게 설계돼서, python3가 없으면 티 안 나게 꺼질 수 있어요. doctor가 hook 4종·스캐너를 실제로 돌려 ✅/❌로 확인해줘요.

정직함이 곧 기능이에요

여기까진 기계가.
여기서부턴 AI가.

”전부 자동!”이라고 뭉뚱그리지 않아요. 기계가 확실히 하는 것과 AI가 알아서 하는 것을 나눠서 보여줘요.
왼쪽은 설치하면 늘 돌아가고, 오른쪽은 Claude가 판단해서 해요.

MACHINE · 훅이 직접 실행

기계가 확실히 하는 것

  • 대표적인 위험 패턴(숨는 에러·비밀키) 경고Claude가 Write/Edit로 코드 쓸 때
  • 대화가 정리되기 전에 진행상황 저장자동 정리·/compact 직전
  • 새로 시작할 때 규칙 다시 챙겨주기새 대화 시작할 때
  • 뭔가 바꿨는데 기록 안 하면 알려주기대화 끝날 때
AI · 필요하면 명령어로

AI가 알아서 하는 것

  • “만들어줘” 하면 이미 있는지 먼저 찾기새로 만들기 전
  • “다 했어” 하면 점검하고 문서도 맞추기다 만든 후
  • “검토해줘” 하면 다른 눈으로 과한 설계 잡기설계 살펴볼 때
  • “고쳐줘” 하면 대충 덮지 말고 근본 원인부터버그·에러 고칠 때
⚠️
hi-vibe는 "버그 자동 탐지기"가 아니에요.
코드를 훑어 버그를 찾아주는 도구가 아니라, AI가 대충 못 하게 만드는 안전벨트예요. 버그를 잡는 건 결국 Claude의 추론이고, hi-vibe는 그 추론을 빡세게 하도록 강제하는 규율·문서·기계 장치예요. "자동으로 다 잡아줘요"가 아니라 "안 하던 규율을 강제로 하게 만들어요" — 기대를 여기 맞추면 실망이 없어요.
💡
🤖 “AI가 알아서”는 100% 보장은 아니에요.
AI가 깜빡할 수도 있어요. 그럴 땐 명령어(/hi-vibe:find 같은 거)를 직접 눌러서 확실히 챙기면 돼요 — AI가 놓친 안전벨트를 손으로 채우는 셈이에요.
🎚️
켜지는 방식이 3단이에요.
🔒 깔면 거저 (훅) — init만 하면 왼쪽 ⚙️ 4개가 자동. 코드 쓸 때마다 에러 삼킴·비밀키를 기계가 잡아요. 진짜 항상 매인 벨트.
🧭 부르면 (find · review · check) — 말하면 도는 🤖 규율·스캔. 강력하지만 AI가 놓치면 안 걸려요. 잔소리하는 코치.
🛡️ 원하면 (gate) — 복잡도·순환의존 린트. 제일 많이 잡지만 침습적이라 준비됐을 때 1회 설치(필수 아님).

명령어 한눈에

대부분 자동. 필요할 때만 직접.

⚡ 자동은 평소처럼 말하면 알아서 걸리고,
🖐 직접은 설치·점검처럼 내가 필요할 때 눌러요.

🔒
설치는 전역 한 번, init은 프로젝트마다.
/plugin install은 기본이 유저 스코프(전역)라 한 번 설치하면 모든 프로젝트에서 /hi-vibe: 명령어를 써요(프로젝트마다 재설치 X).
하지만 자동 기능(훅)은 쓰고 싶은 프로젝트마다, 그 최상위 폴더에서 /hi-vibe:init을 해야 켜져요
— .hi-vibe/ 마커가 만들어진 폴더에서만 돌아요.
init 안 한 다른 프로젝트엔 전혀 개입하지 않아요
— 전역 설치인데도 원하지 않는 곳에서 훅이 도는 걸 막는 안전장치예요.
(새 프로젝트에서 쓰려면 그 폴더에서 /hi-vibe:init 한 번.)

🚀 상황별 빠른 시작 — 그래서 뭘 치면 돼요?

설명 다 안 읽어도 돼요. 지금 내 상황을 찾아 그 명령어만 치세요. (평소엔 그냥 말하면 자동으로 걸려요.)

내 컴퓨터에 방금 설치
/hi-vibe:doctor → /hi-vibe:init (잘 켜졌나 확인 → 이 프로젝트에 켜기)
이미 만들던 프로젝트에 설치
init → doctor → review --all --deep (지금까지 쌓인 코드를 처음 보는 클로드 눈으로 전체 설계 검토 — 강력 추천)
새 기능 만들 때
그냥 평소처럼 말하기 (자동으로 “이미 있나” 먼저 찾아요) · 확실히 하려면 /hi-vibe:find

더 자세히 — 헷갈리기 쉬운 것들 (펼쳐 보기)

🔍 review는 하나에 옵션 두 개 (--all · --deep)
review는 방금 만든 기능 하나를, review --all은 세션에서 바뀐 기능 전체를 한 번에 검증해요. (안 바뀐 건 자동 skip — 여러 개 몰아 만든 뒤 “전체 리뷰해줘”에 좋아요.)
review --deep은 코드를 짠 기억이 없는 서브에이전트를 새로 불러와, 편견 없이 깨끗한 눈(fresh-eyes)으로 과잉설계·숨은 결합을 봐요 — 새로운 시각으로 설계를 검증하는 거예요.
정리하면 --all은 모든 기능 전체를, --deep은 새 서브에이전트가 검증해요. 서로 별개의 축이라 review --all --deep도 돼요.
🧭 check vs review — 헷갈리기 쉬운데 완전 달라요
check는 기계(스캐너)가 저장소 구조를 훑어 “뭐가 있는지”를 증거로 짚어요 — 중복 코드·안 쓰는 코드·너무 큰 파일. 사실 확인이에요.
review --deep은 서브에이전트가 방금 바꾼 코드를 설계 관점으로 봐 “잘 만들었는지”를 판단해요 — 과잉설계·더 단순한 길·숨은 결합. 의견·판단이에요.
한 줄로: check = 뭐가 있는지(사실), review = 잘 만들었는지(판단). 겹치는 게 아니라 서로 보완이에요.
🔁 check와 gate는 리듬이 반대예요
check는 내가 돌리는 진단 — 구조가 궁금할 때마다 몇 번이든 반복해요.
gate --ci는 프로젝트마다 한 번 깔면 끝인 관문 — 설치는 1회, 이후 push마다 GitHub이 자동 검문(다시 칠 필요 없어요).

뒤에서 뭐가 도나 — 스킬(엔진) ↔ 명령어(버튼)

쉬운 명령어는 버튼일 뿐, 실제 일은 뒤의 스킬(엔진)이 해요.
같은 엔진을 명령어로도, "평소 말"로도, AI가 알아서도 불러요.

What it catches

Let AI write your code and this keeps happening.

It builds fast — but as the project grows, small mistakes quietly pile up.

Rebuilds features that already existIt forgets code it made before and writes something similar from scratch. (e.g. the same function / helper again)
Hides errors instead of fixing themIt patches over the cause instead of solving it, so it blows up bigger later. (e.g. papering over with a fallback / default)
Decides on its own without checkingIt guesses on ambiguous requirements or stale facts instead of verifying. (e.g. API limits · pricing · versions)
Code gets harder and harder to touchIt keeps piling features into one place until editing feels scary. (e.g. skipped edge cases · god files)

How · three layers

Blocked by structure, not by asking nicely.

The seatbelt is woven in three layers — gentle up top, machine-enforced at the bottom.

Layer one

Remembers for you

It auto-records what the project is, how far you got, and what changed — so a fresh chat never loses the thread.

Layer two

Teaches the AI the rules

Rules like “search before you build”, “fix the cause, not the symptom”, “don’t state guesses as fact” are planted into the AI so it polices itself.

Layer three

The machine blocks it

A code checker catches risky code on the spot — and if you want, it can fail the build before it ever reaches GitHub.

Feature by feature

Install it, and your project gets this.

The core idea: docs aren't dumped in one file; they're split across the project and auto-updated as the code changes.

Prevention⚡ AUTO
🔍No duplicate reimplementationfind

Before making a new function or file, it checks if one already exists — so the AI stops rebuilding last session's work. When there's none, it says "not found across N scanned files" — evidence, not a guess.

🩹No band-aid error handlinginstant detect

Instead of burying an error under a fallback, it makes you look at the cause. Code that silently swallows errors (empty catch, except: pass) is flagged the moment you write it. One comment lets it pass if intended.

📊No unfounded claimsdiscipline

It won't state "probably X" as fact. Numbers and APIs get verified — or labeled "estimate" when they can't be. (With context7, it pulls the latest docs.)

Memory & review⚡ AUTO
👀Fresh-eyes design reviewreview --deep

A fresh Claude with zero memory of writing the code reviews it — with no bias. It catches what a checklist can't: over-engineering, needless features, hidden coupling — "this looks like too much."

🧠Memory, handover & recallhandover · recall

Auto-records progress every time the chat is compacted, and re-injects the latest handover on a new chat. Ask "why did we do it this way?" and it digs the records for the date and reason.

🎯No false positivesdiscipline · scanner

It won't wrongly call live code "dead." Framework-registered functions, tests, export default components, and work-in-progress code are kept off the dead list by a false-positive index.

Manual checks🖐 MANUAL
🔬Structure checkupcheck

Scans a code-heavy project for not just exact duplicates but "90%-reimplemented" functions, unused code, and oversized files. It tells work-in-progress apart before suggesting a delete.

🛡️Machine enforcement (lint · CI)gate --ci

gate installs the checkers (ruff·eslint, types, circular deps) locally — your editor flags them. gate --ci gates every push — violations fail the build.

🩺Self-diagnosisdoctor

Hooks are built to fail silently so they never block Claude Code — so a missing python3 could turn them off unnoticed. doctor actually runs all 4 hooks and the scanner, showing ✅/❌.

hi-vibe is not a "bug detector."
It doesn't scan your code to find bugs; it's a seatbelt that stops the AI from cutting corners. Catching bugs is still Claude's reasoning — hi-vibe is the discipline, docs, and machinery that force that reasoning to be rigorous. Not "it auto-catches everything" but "it forces the discipline you'd otherwise skip." Set expectations there and you won't be let down.

The AI can forget. When you want to be sure, run the command (like /hi-vibe:find) yourself — the manual latch for the seatbelt the AI missed.

More detail — easy-to-confuse things (expand)

🔍 One review, two options (--all · --deep)
review checks the one feature you just made; review --all checks every feature changed this session at once (already-reviewed, unchanged files are skipped — great after building several things, then “review everything”).
review --deep spawns a new subagent that never wrote this code, so with no bias it uses clean eyes (fresh-eyes) to catch over-engineering and hidden coupling the author can't see.
In short: --all is how wide (one→all), --deep is who reviews (author→a fresh subagent) — different axes, so review --all --deep works too.
🧭 check vs review — easy to confuse, but totally different
check is a machine (scanner) sweeping your repo's structure to point out “what's there” with evidence — duplicate code, unused code, oversized files. It's fact-finding.
review --deep is a subagent looking at what you just changed from a design angle to judge “is it well-built?” — over-engineering, simpler alternatives, hidden coupling. It's judgment.
In one line: check = what exists (facts), review = is it good (judgment). They complement, not overlap.
🔁 check and gate have opposite rhythms
check is a diagnostic you run — repeat it whenever you're curious about structure.
gate --ci is a guard you install once per project — after that GitHub checks every push automatically, no need to re-run it.
